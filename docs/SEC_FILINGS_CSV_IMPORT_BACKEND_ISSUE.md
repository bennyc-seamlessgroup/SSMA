# SEC Filings CSV Import Backend Issue

> **Contract update (July 20, 2026):** The revised integration specification defines the canonical record-array object as `manual-input/sec-filings/{ticker}/sec-filings.json` and adds `inputRows`, `importedRows`, `skippedRows`, and `errors` to the import response. The portal now validates this contract. The `/None` path below documents the original failure that the backend fix must prevent.

## Executive Summary

`POST /manual-input/import` reports a successful SEC filings import for ticker `MIMI`, but writes the generated data to an invalid object key:

```text
manual-input/sec-filings/MIMI/None
```

The normal read endpoint continues to read:

```text
manual-input/sec-filings/MIMI/sec-filings.json
```

As a result, the imported MIMI records are not returned by `GET /manual-input/sec-filings?ticker=MIMI`. The frontend continues to display the old data returned by that endpoint. This is a backend storage-key resolution problem, not a frontend caching or consolidation problem.

The import response also reports 68 imported records, but the supplied CSV contains one header row and 67 data rows. The CSV header may be incorrectly included in `recordsCount`.

## Severity and Impact

- **Severity:** High for manual data operations
- **Affected workflow:** Operations Portal > Data Import > SEC filings
- **Affected category:** `sec-filings`
- **Observed ticker:** `MIMI`
- **User impact:** The UI reports success, but the imported records are not available through the normal API and do not appear in the user portal.
- **Data risk:** Repeated imports can create orphan objects while leaving stale production data unchanged.

## Observed Result

The import reports success and returns a generated object resembling:

```json
{
  "message": "Import completed successfully",
  "category": "sec-filings",
  "ticker": "MIMI",
  "recordsCount": 68,
  "generatedFiles": [
    "manual-input/sec-filings/MIMI/None"
  ]
}
```

After a successful response, `GET /manual-input/sec-filings?ticker=MIMI` still returns the old records rather than the uploaded MIMI dataset.

## Expected Result

The importer should replace the category's canonical object:

```text
manual-input/sec-filings/MIMI/sec-filings.json
```

The response should resemble:

```json
{
  "message": "Import completed successfully",
  "category": "sec-filings",
  "ticker": "MIMI",
  "recordsCount": 67,
  "generatedFiles": [
    "manual-input/sec-filings/MIMI/sec-filings.json"
  ]
}
```

Immediately afterward:

```http
GET /manual-input/sec-filings?ticker=MIMI
```

must return the 67 imported MIMI records.

## Likely Root Cause

The category configuration or filename resolver for `sec-filings` appears to return a null value that is converted to the string `None` when the S3 key is assembled.

Backend areas to inspect:

1. The category registry used by `POST /manual-input/import`.
2. Hyphenated versus underscored category normalization: `sec-filings` versus `sec_filings`.
3. The configured output filename for record-array imports.
4. The function that builds `generatedFiles` and the S3 destination key.
5. Whether the writer accepts a missing filename instead of rejecting the request.
6. Whether import and GET use different category registries or key-building functions.
7. Whether the CSV parser counts the header as a data record.

The normal GET path and the import output path must be generated by the same canonical category definition.

## Required Backend Fix

### 1. Define a Canonical Category Configuration

The exact implementation can follow the backend's existing conventions, but the effective configuration should be equivalent to:

```json
{
  "sec-filings": {
    "storageType": "record-array",
    "prefixTemplate": "manual-input/sec-filings/{ticker}/",
    "objectName": "sec-filings.json",
    "writeMode": "replace"
  }
}
```

### 2. Use One Key Resolver for Import and Read

Both operations must resolve to the same key:

```text
manual-input/sec-filings/{NORMALIZED_TICKER}/sec-filings.json
```

Do not maintain separate hardcoded path logic for import and GET.

### 3. Reject Missing Object Names

Never write a file if the resolved object name is null, empty, `None`, `null`, or `undefined`.

Return a server configuration error before writing:

```json
{
  "error": "No output object name is configured for category sec-filings"
}
```

The importer must not return HTTP 200 when the generated target key is invalid.

### 4. Correct CSV Record Counting

`recordsCount` must count parsed data rows only. It must not include:

- The header row
- Empty rows
- Rows rejected by validation

For the supplied file, the expected count is 67.

If rows are skipped, return explicit counts:

```json
{
  "inputRows": 67,
  "importedRows": 67,
  "skippedRows": 0,
  "errors": []
}
```

### 5. Validate the Ticker

Normalize the requested ticker to uppercase. If `recordTicker` is present in the CSV, every non-empty value should match the requested ticker.

For example:

```text
Request ticker: MIMI
CSV recordTicker: MIMI
Result: valid
```

A mismatch should return HTTP 400 and must not modify the existing object.

### 6. Validate Before Replacing Existing Data

Recommended write sequence:

1. Parse the complete CSV.
2. Validate headers and every record.
3. Verify the ticker and record count.
4. Serialize the complete output.
5. Write to a temporary object or staged key.
6. Replace the canonical object only after validation succeeds.
7. Return the final canonical key in `generatedFiles`.

This prevents a malformed upload from deleting or partially replacing valid SEC filings data.

## Data Contract

Each imported record should support the following fields. Backend-generated metadata may be added after validation.

| Field | Required | Type | Notes |
|---|---:|---|---|
| `id` | Yes | string | Preserve a valid supplied ID or generate a stable unique ID. |
| `ticker` | Yes | string | Normalize from `recordTicker` or the request ticker. Expected `MIMI`. |
| `companyName` | Yes | string | Company display name from CSV. |
| `formType` | Yes | string | SEC form type such as `6-K`. |
| `formDescription` | No | string | Human-readable filing description. |
| `filingDate` | Yes | `YYYY-MM-DD` | Validate as an ISO date. |
| `reportingDate` | No | `YYYY-MM-DD` | Empty values may be null. |
| `act` | No | string | Preserve leading zeros if applicable. |
| `filmNumber` | No | string | Treat as text, not a number. |
| `fileNumber` | No | string | Treat as text to preserve formatting. |
| `accessionNumber` | No | string | Preserve dashes and leading zeros. |
| `filingsUrl` | No | string | Validate as an HTTP or HTTPS URL when present. |
| `notes` | No | string | Empty string or null is acceptable. |
| `createdBy` | Backend | string | Authenticated user identity when supported. |
| `createdAt` | Backend | ISO timestamp | Import timestamp. |
| `updatedBy` | Backend | string | Authenticated user identity when supported. |
| `updatedAt` | Backend | ISO timestamp | Import timestamp. |

The source column `tradeDate` is not currently part of the SEC filings API record contract. The backend should either document its intended mapping or explicitly ignore it. It must not shift the remaining columns during parsing.

## Consolidation Behavior

Consolidation should only run after a valid import has written the canonical object. It cannot repair an import written to:

```text
manual-input/sec-filings/MIMI/None
```

If SEC filings are consumed by a downstream consolidated output, the sequence should be:

1. Import and validate CSV.
2. Write `sec-filings.json`.
3. Confirm the canonical object can be read.
4. Trigger consolidation if required by the pipeline.
5. Return the consolidation status separately from the import status.

Do not report a successful import solely because a noncanonical S3 object was created.

## Cleanup Required

After deploying the fix:

1. Locate orphan objects under `manual-input/sec-filings/*/None`.
2. Confirm whether they contain recoverable imports.
3. Re-import or migrate valid data to `sec-filings.json`.
4. Delete the orphan `None` objects only after the canonical data is verified.
5. Do not delete an existing valid `sec-filings.json` until its replacement has passed validation.

## Acceptance Tests

### Successful Import

- Upload the supplied MIMI CSV.
- API returns HTTP 200.
- `recordsCount` equals 67.
- `generatedFiles` contains exactly `manual-input/sec-filings/MIMI/sec-filings.json`.
- No generated path contains `None`, `null`, `undefined`, or an empty filename.
- GET immediately returns 67 MIMI records.
- Every returned record has ticker `MIMI`.
- CURR data remains unchanged.

### Replacement Behavior

- Import the same CSV twice.
- The second import replaces the category dataset rather than appending duplicates.
- GET still returns 67 records, not 134.

### Ticker Isolation

- Import MIMI data while CURR data exists.
- MIMI GET returns only MIMI data.
- CURR GET returns only CURR data.
- Neither ticker overwrites the other ticker's object.

### Validation Failure

- Upload a CSV whose `recordTicker` differs from the request ticker.
- API returns HTTP 400.
- Existing canonical data is unchanged.
- No orphan or temporary object remains.

### Missing Category Configuration

- Simulate an unconfigured output object name.
- API returns an error.
- No object named `None`, `null`, or `undefined` is written.

### Header and Blank Rows

- Confirm the header is not counted as a record.
- Confirm trailing blank lines are not counted.
- Confirm validation errors identify the source row number.

## Regression Coverage

The same shared importer may serve other manual-input categories. Test all record-array categories for the same filename-resolution issue, especially categories whose API name contains a hyphen.

For every supported category, verify:

- Import and GET use the same canonical key builder.
- The configured object name is non-empty.
- The response reports the actual final object key.
- Record counts exclude headers and blank lines.
- Ticker isolation works.
- Repeat imports follow the documented replace or append policy.
