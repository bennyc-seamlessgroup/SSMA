# DATA_STRUCTURE.md

Implementation specification for restructuring Currenc Intelligence portal data outputs from raw vendor/manual feeds into scalable JSON contracts. This document is generated from `newest datapoint csv/Portal Data Point - Master Table (1).csv` and the base samples in `Justin json-file/`.

## Executive Summary

The current portal receives many separate JSON files that mix raw vendor records, page-ready metrics, manual inputs, and derived calculations. The recommended redesign is a **hybrid domain-based architecture**: keep append-only historical datasets by domain, keep latest page-ready snapshots as small current JSON files, and keep manual inputs/audit logs separate from vendor data. Pages should compose these domain files rather than own their own duplicated page JSON.

Recommended top-level principles:
- Preserve raw vendor pulls in immutable dated archives for traceability.
- Generate normalized historical JSON arrays for time-series charts and tables.
- Generate current snapshot JSON objects for KPI cards and page summaries.
- Store operations inputs and user inputs separately with creator, updater, timestamps, workspace/ticker, and audit history.
- Store AI/rule-engine text separately from numeric metrics so model-generated content can be refreshed independently.
- Version every JSON schema with `schemaVersion`, `ticker`, `generatedAt`, and `sourceWatermarks`.

## Proposed Architecture

### Option A: One Large Unified JSON

Pros: one fetch can hydrate the whole portal; simple for prototypes; easy to inspect manually.

Cons: poor caching granularity; large payloads; one update invalidates every page; high merge conflict risk in backend jobs; hard to version domains independently; frontend must parse unrelated data; not scalable for multiple tickers and long histories.

Recommendation: **Do not use** except for local demos.

### Option B: One JSON Per Portal Page

Pros: simple page consumption; small mental model for frontend; each page can load one file.

Cons: duplicates fields such as borrow fee, utilization, issued share, price, social counts, and ownership values across pages; creates inconsistent values if jobs run at different times; backend must know page layouts; hard to reuse data for reports and alerts.

Recommendation: useful only for backward-compatible page adapters, not as the canonical backend structure.

### Option C: Hybrid Domain-Based JSON (Recommended)

Separate data by domain and lifecycle:
- `raw/{vendor}/{ticker}/{date}/...` immutable vendor payloads.
- `history/{ticker}/...` append-only normalized time series and event arrays.
- `current/{ticker}/...` latest snapshot objects optimized for KPI cards and summary panels.
- `manual/{ticker}/...` operations and customer input state plus audit logs.
- `analysis/{ticker}/...` AI and rule-engine outputs.
- `reports/{ticker}/...` report index and report payloads.

This gives clean ownership, efficient cache invalidation, lower frontend payloads, and reusable data for pages, alerts, reports, and AI analysis.

## Historical JSON Specifications

### `history/{ticker}/market-history.json`

Purpose: daily market, short interest, borrow fee, availability, margin, utilization, days-to-cover, and price data used by Dashboard, Short Interest, and Lending Pressure charts/cards.

- Primary key: `ticker + tradeDate`
- Date field: `tradeDate` (`YYYY-MM-DD`, exchange/session date)
- Update frequency: daily next-day for most vendor/manual fields; price may update intraday if live feed is licensed.
- Sources: Chart Exchange, IBKR/FUTU/operations input, backend calculations.
- Append rule: upsert by `tradeDate`; keep `createdAt`, `updatedAt`, and source-specific timestamps.

Schema:

```json
{
  "records": [
    {
      "tradeDate": "2026-06-18",
      "issuedShare": 112280000,
      "price": 3.08,
      "borrowFeePercent": 29.163,
      "availableSharesChartExchange": 4000000,
      "availableSharesIbkr": null,
      "availableSharesFutu": null,
      "availableShares": 4000000,
      "utilizationPercent": null,
      "daysToCover": 5.36,
      "shortInterestShares": 947482,
      "shortInterestPercent": 8.97,
      "shortScore": null,
      "initialMarginIbkr": null,
      "initialMarginFutu": null,
      "initialMargin": null,
      "maintenanceMarginIbkr": null,
      "maintenanceMarginFutu": null,
      "maintenanceMargin": null,
      "averageDurationDays": null
    }
  ]
}
```

### `history/{ticker}/short-volume-history.json`

- Primary key: `ticker + date`
- Date field: `date`
- Source: Chart Exchange short volume (`chartexchange_short_volume_*`)
- Update frequency: daily next-day
- Portal: Short Interest page Short Volume table

Schema should normalize exchange codes but retain source values: `totalVolumeReported`, `totalShortVolumeReported`, `totalLongVolumeReported`, `offExchangeNonExempt`, `offExchangeExempt`, `nasdaqBx`, `nasdaqPhlx`, `nyse`, `nyseArca`, `nyseNational`, `nyseAmerican`, `chx`, `cboeEdgx`, `cboeBzx`, `cboeEdga`, `cboeByx`.

### `history/{ticker}/ftd-history.json`

- Primary key: `ticker + settlementDate`
- Date fields: `settlementDate`, `tradeDate`, `closingDeadline`
- Source: Chart Exchange failure-to-deliver (`chartexchange_failure_to_deliver_*`)
- Update frequency: when FTD dataset updates; expect SEC lag
- Portal: Short Interest page FTD table
- Base sample was empty; backend must still generate an empty-but-valid schema.

### `history/{ticker}/ownership-history.json`

- Primary key: `ticker + holderName + formType + fileDate + effectiveDate + sourceType`
- Date fields: `fileDate`, `effectiveDate`
- Sources: Fintel security ownership and activist filings
- Update frequency: daily ingest, but 13F/major-holder data is naturally quarterly/event-driven
- Rules: filter institutional security ownership by configured `IO_security_name`; ignore option/call records (`putCall == "Call"` or option/call equivalent).
- Portal: Ownership tables, institution breakdown, insider suggestions for Internal Float.

### `history/{ticker}/sentiment-events.json`

- Primary key: stable hash of `platform + link + datetime + author + content`
- Date field: `datetime` ISO string
- Sources: social-data S3 prefixes for Reddit/X/Facebook/LinkedIn; operations upload for Stocktwits
- Update frequency: near-real-time or batch depending source
- Portal: Social Sentiment feed, platform tabs, sentiment timeline, platform breakdown.
- Normalize `Twitter` to platform display `X`; normalize `Linkin` typo to `LinkedIn`.

### `history/{ticker}/sec-filings-history.json`

- Primary key: `accessionNumber`; fallback `ticker + formType + filingDate + filingsUrl`
- Date fields: `filingDate`, `reportingDate`
- Source: operations input or future SEC API
- Update frequency: event-driven/manual
- Portal: SEC Filings page and Dashboard filing-event markers.

## API Recommendations

Use backend APIs as the canonical writer and JSON generation jobs as readers/materializers.

Recommended read APIs:
- `GET /companies/{ticker}/profile`
- `GET /market-data/{ticker}/current`
- `GET /market-data/{ticker}/history?start=&end=&fields=`
- `GET /ownership/{ticker}/current`
- `GET /ownership/{ticker}/history?source=institutions|insiders`
- `GET /internal-float/{ticker}/current`
- `GET /sentiment/{ticker}/current?period=1D|1W|1M|6M|1Y`
- `GET /sentiment/{ticker}/events?platform=&start=&end=&cursor=`
- `GET /sec-filings/{ticker}?cursor=&limit=`
- `GET /reports/{ticker}/index`

Recommended write APIs for operations/customer-maintained fields:
- `POST /market-data/{ticker}` and `POST /market-data/{ticker}/batch`
- `POST /management-holdings`, `PATCH /management-holdings/{id}`, `DELETE /management-holdings/{id}`
- `POST /internal-float/{ticker}/inputs`, `PATCH /internal-float/{ticker}/inputs/{id}`, `DELETE /internal-float/{ticker}/inputs/{id}`
- `POST /sec-filings`, `PUT /sec-filings`, `DELETE /sec-filings/{accessionNumber}`
- `POST /hotkeys`, `DELETE /hotkeys/{ticker}/{kwatchHotkey}`
- `POST /sentiment/stocktwits/batch`

Every write response should return the saved record, current materialized summary if cheap, and `auditId`.

## Field-Level Rules and Formulas

- Changes: `numChange = current - previousComparableRecord`; `percentChange = numChange / previousComparableRecord * 100`; if denominator is zero/null, return `null` plus display label `No prior period`.
- Short score risk factor: `<=25 Low`, `<=50 Moderate`, `<=75 High`, `>75 Extreme` unless business changes thresholds.
- Borrow fee risk factor: `<25 Low`, `<50 Moderate`, `<75 High`, `>=75 Extreme`.
- Days to cover risk factor: `<1 Low`, `<3 Moderate`, `<7 High`, `>=7 Extreme`.
- Available shares/utilization/lending pressure risk factors currently inherit borrow-fee risk factor per CSV; this should be reviewed because it may oversimplify risk.
- Available shares: `MAX(available_shares_chart_exchange, available_shares_ibkr, available_shares_futu)`.
- Initial margin: `MAX(initial_margin_ibkr, initial_margin_futu)`.
- Maintenance margin: `MAX(maintenance_margin_ibkr, maintenance_margin_futu)`.
- Institutional owners count: count active Fintel security ownership records after security name filter and excluding calls/options.
- Institutional shares long: `SUM(owners.shares)` after filters.
- Institutional holding percent: `institutionalSharesLong / issuedShare * 100`.
- Ownership public float: `issuedShare - totalStrategicEntities - totalInstitutions` per current ownership page direction; do not expose internal customer-only token/collateral deductions on public Ownership page.
- Internal Float real tradable float: `issuedShare - institutionalSharesLong - approvedManagementStrategicHoldings - tokenizedShares - collateralizedShares` if tokenized/collateralized are confirmed as non-tradable deductions. Confirm with business if institutions should be subtracted before or after internal adjustments.
- Sentiment score: average of record sentiment values where positive/bullish=100, neutral=50, negative/bearish=0.
- Sentiment ratios: `positiveCount / totalCount * 100`, `neutralCount / totalCount * 100`, `negativeCount / totalCount * 100`.
- Sentiment platform contribution: `platformRecordCount / totalRecordCount * 100`.

## Fields Missing From Base JSON

## Performance Recommendations

- Lazy-load domain files by route. Dashboard should load `market-current`, `market-history` for selected range, `sec-filings-history` event markers, and alert state only; it should not load full ownership or full sentiment feeds.
- Cache current snapshots for 30-60 seconds. Cache historical arrays longer because prior dates rarely change; invalidate only when a new source watermark appears.
- Split long histories by year once arrays exceed practical browser size, e.g. `history/{ticker}/market-history/2026.json`. Provide an index file listing available years and watermarks.
- Compress JSON at CDN level with Brotli/Gzip. Do not manually minify at the cost of debuggability in source storage; let CDN handle transfer compression.
- Use cursor pagination for social events, ownership history, SEC filings, short-volume, and FTD tables.
- Store raw vendor files immutable and deduplicate normalized history by primary key. Never append duplicate Fintel rows from repeated daily pulls.
- Materialize current snapshots after every ingest so the frontend avoids heavy calculations.
- Include `sourceWatermarks` per file so red-dot update detection compares data content, not only modified timestamps.
- Use incremental update files for social feed and SEC filings if near-real-time volume grows, e.g. `sentiment-events/YYYY/MM/DD/HH.json`.

## Final Recommended Architecture

Use the hybrid domain architecture. Backend jobs should ingest raw vendor/manual data, normalize into append-only historical domain files, compute small current snapshots, and expose CRUD APIs for manual input. Page-specific JSON files may remain temporarily as compatibility adapters, but they should be generated from domain files and eventually removed.

Implementation order:
1. Create normalized `history/{ticker}` files for market, ownership, sentiment, SEC filings, short volume, and FTD.
2. Create `manual/{ticker}` input stores and audit logs for operations and customer-maintained fields.
3. Create current snapshot generators for `market-current`, `ownership-current`, `internal-float-current`, and `sentiment-current`.
4. Move AI/rule-engine outputs into `ai-analysis-current`.
5. Add API endpoints and make JSON outputs a cache/materialization layer, not the primary write surface.
6. Keep backward-compatible page JSON adapters only while frontend migration is in progress.
